# Epic Writing Skill

An epic document is a planning artifact for large, multi-phase projects. Unlike a PRD (single feature or fix), an epic spans multiple PRDs, takes months to years, and requires a decision-maker to commit to a long-term direction.

The five-section structure:

  # Problem
  # Background
  # Impact
  ---
  # End Picture
  # Roadmap

---

## 핵심 원칙 (모든 섹션에 적용)

**prd-writer 스킬의 모든 문체·용어 원칙이 그대로 적용됩니다.**
- ~합니다 / ~습니다 체(경어체) 일관 사용
- 확신도에 따른 어미 구분: 사실 → ~합니다, 가능성 → ~수 있습니다
- 영어 용어는 영어 그대로, 괄호 병기는 최초 1회만
- 구체성: 막연한 문장 → 구체적 동작·이름·숫자로 교체
- 모르면 솔직하게: `** 확인 필요`로 인라인 표기

**DARO 공유 컨텍스트는 prd-writer 스킬을 참조합니다.** Background에서 재설명하지 않습니다.

---

## PRD vs Epic 차이

| | PRD | Epic |
|---|---|---|
| 범위 | 단일 기능·수정 | 다단계 장기 프로젝트 |
| 주요 독자 | 담당 엔지니어 | 의사결정권자 + 개발팀 |
| 기간 | 1~4주 | 수개월 ~ 수년 |
| 결과물 | 구현 명세 | 방향성 + 단계 계획 + PRD 목록 |
| Solution 섹션 | 구현 상세 포함 | 없음 (Roadmap + PRD 링크로 대체) |
| End Picture | 없음 | **필수** — 완전 완료 시의 비전 |

---

## Stage 1: Context Gathering

prd-writer의 맥락 수집(질문 1~6)에 추가로 파악합니다:

1. **End Picture 시점은 언제인가?** (몇 년 후를 보고 있는가)
2. **주요 의사결정권자는 누구인가?** (이 문서로 누구를 설득해야 하는가)
3. **현재 어느 단계인가?** (MVP 전, MVP 중, 첫 Phase 완료 후 등)
4. **각 Phase의 성공 기준이 있는가?**
5. **이미 작성된 연결 PRD가 있는가?**

**Background 해상도를 높이기 위해 추가로 파악합니다:**

6. **이 문제가 발생하는 구조적 메커니즘은 무엇인가?** "현재 구조가 없다"가 아니라, 왜 그런 구조가 만들어졌는지 — Background 첫 줄의 재료
7. **독자 demand/기능 확보의 대안들이 있는가?** 있다면 왜 그것이 충분하지 않은가 — 이 Epic 방향을 선택한 근거
8. **지금 당장 존재하는 구체적인 사례나 파트너가 있는가?** Background에 구체성을 부여하고 Solution 출발점을 명확히 하는 재료

---

## Stage 2: 섹션별 작성 가이드

### Problem

**prd-writer의 Purpose와 동일한 기준.** 1문장, 절 2개 이하.

Epic이므로 개별 기능 결함보다 **구조적·전략적 문제**를 다룹니다.

좋은 예:
> "DARO의 demand 전체가 외부 Demand·미디에이션 네트워크에 의존하며, 독자 demand 없이는 퍼블리셔에게 차별적 eCPM 경쟁력을 스스로 만들어낼 수 없습니다."

---

### Background

**prd-writer의 Background 규칙과 동일.** 5줄 이내, 상세는 토글.

**논리 흐름 패턴 (Epic 권장 4줄 구조):**

```
줄 1-2: [메커니즘 + 방향] — 현재 구조가 왜 이 문제를 만드는가, 어떤 종류의 것이 필요한가

줄 3: [선택의 근거] — 여러 대안 중 왜 이 Epic의 방향인가

줄 4: [구체적 기회] — 지금 당장 존재하는 기회·사례·파트너
```

각 줄은 이전 줄에서 논리적으로 이어져야 합니다. 마지막 줄은 Roadmap의 Phase 1으로 자연스럽게 연결되어야 합니다.

**줄바꿈으로 호흡을 만들 것**: 논리적 묶음(메커니즘 / 선택 근거 / 기회) 사이에 줄바꿈을 넣어 독자가 어느 지점에서 끊어 읽어야 하는지 알 수 있게 합니다. 4줄을 한 덩어리로 붙여 쓰면 읽는 호흡이 불명확해집니다.

**선택지 나열 시 ①②③ 번호 사용**: 대안을 여러 개 나열할 때 번호를 붙이면 개수와 구조가 즉시 파악됩니다.

**구체적 출발점은 "기회 서사"로, 사실 나열로 끝내지 않을 것**:
- ❌ "번개장터가 이미 DARO 퍼블리셔로 연동되어 있으며, 내부 판매자 광고 수요가 출발점이 됩니다." (현황 나열 — 의사결정자에게 왜 지금인지가 전달되지 않음)
- ✅ "번개장터가 내부 판매자 광고 운영에 대한 니즈가 있는 만큼, DSP 초기 모델을 시도해볼 수 있는 좋은 상황이 준비되어 있습니다." (기회 프레이밍 — 지금이 타이밍임을 전달)

**플라이휠 역할 전환이 있으면 명시**: 기존 관계자가 새 역할(예: 퍼블리셔 → 광고주)로 전환될 수 있는 구조라면 Background에서 명시합니다. 이 역할 전환이 End Picture와 Impact의 핵심 논리를 미리 예고하며, 독자가 나중에 "왜 이게 가능한가"를 되짚지 않아도 되게 만듭니다.

피할 것:
- **단순 정보 나열**: 독자가 "그래서 왜 이 방향인가"를 스스로 추론해야 한다면 논리 흐름이 부족한 것입니다.
- **구체적 구현 방안 제시**: "~기능을 만들어야 합니다" 같은 구현 지시는 금지. 단, "어떤 종류의 것이 필요하다"는 방향 언급은 허용.
- DARO 공유 컨텍스트 재설명

---

### Impact

**"이 Epic이 완료되면 어떤 비즈니스 변화가 생기는가."**

**Impact가 End Picture 앞에 오는 이유**: Impact는 압축적인 비즈니스 변화이고, End Picture는 상세한 제품 비전입니다. 독자가 "왜 중요한가"를 먼저 파악한 뒤 상세 비전으로 들어가도록 유도합니다. 문서에서는 두 섹션 사이에 `---` 구분선을 넣어 레이어가 다름을 시각적으로 명확히 합니다.

prd-writer의 Impact와 동일한 기준이지만, **전략적 수준의 변화**를 다룹니다.

- 숫자가 있으면 AS-IS / TO-BE 테이블
- 숫자가 없으면 bullet 3개 이하, 각 1문장
- "기능적 변화" / "비즈니스 변화" 소헤더 없음
- Solution 기능이나 End Picture 비전을 언급하지 않음 — Impact는 오직 "문제가 없어진 상태"

---

### End Picture

**Epic의 핵심 섹션.** 이 섹션의 퀄리티가 의사결정권자의 장기 투자 결정을 좌우합니다.

> **End Picture ≠ Impact**
> Impact는 "문제가 해결된 상태"입니다.
> End Picture는 "이 Epic이 완전히 완료됐을 때 어떤 제품이 존재하고 어떤 시장에서 어떤 고객에게 무엇을 주고 있는가" — 완성된 제품 비전입니다.

**기준**: 의사결정권자가 이 그림을 읽고 "이걸 달성하기 위해 몇 년이 걸려도 기다릴 수 있다"는 확신이 드는 수준.

**설득 구조 (읽는 순서 = 설득 순서):**

```
비전 서술 → 시장 포지션 → 타겟 고객군(왜 이 타겟) → 핵심 가치(니즈→가치→기능) → 핵심 지표(왜+수치) → 세일즈/운영 구조
```

이 순서는 "어디서 → 누구에게 → 무엇을 → 얼마나 → 어떻게"의 흐름으로, 의사결정자가 투자 근거를 단계적으로 쌓아가는 논리 구조입니다.

**End Picture 작성 규칙:**

**시장 포지션 — 벤치마크 비유 활용**: 이 시장에서 지향하는 포지션과 유사한 성공 사례가 있다면 비유로 명시합니다. 추상적 표현보다 독자에게 즉각적인 레퍼런스를 제공합니다.
- ✅ "국내 논게임 앱 생태계에서 퍼블리셔와 광고주를 동시에 연결하는 유일한 플랫폼 (e.g. 게임시장의 앱러빈 포지션)"
- ❌ "국내 논게임 앱 생태계에서 유일한 closed-loop 플랫폼" (비유 없이 추상적으로만)

**타겟 고객군 — End Picture 전체 세그먼트 기술, MVP 범위는 callout으로 분리**:
- End Picture의 타겟은 최종 완성 상태의 전체 세그먼트를 모두 기술합니다 (MVP 범위로 제한하지 않음)
- MVP 단계의 타겟 범위가 더 좁다면 callout으로 분리해 End Picture 비전과 혼동을 막습니다
- 예: ⚠️ callout "MVP: 초기 단계에서는 [좁은 범위]만 진행합니다."

**핵심 가치 — 타겟 세그먼트와 1:1 대응**:
- 핵심 가치 bullet 수는 타겟 고객군 세그먼트 수와 일치시킵니다
- 각 bullet: "세그먼트의 니즈(~가 필요합니다) → 우리가 제공하는 가치(~를 제공합니다)" 순서
- 핵심 기능 목록은 callout으로 분리합니다 (가치를 제공하기 위한 수단 — 본문 흐름 방해 방지)

**핵심 지표 — 타겟 세그먼트를 모두 커버**:
- 타겟 고객군에서 정의한 세그먼트가 모두 지표에 반영되어야 합니다
- 세그먼트별 별도 행이 아니라 하나의 지표 안에 포함시켜도 됩니다
  - ✅ "DSP 활성 고객사 수 | 0 | T1 3사 + 에이전시 1사 이상" (두 세그먼트 모두 반영)
  - ❌ "DSP 활성 고객사 수 | 0 | 3사 이상" (에이전시 세그먼트 미반영)
- 지표 산출 근거(왜 이 수치인가)는 callout으로 분리합니다

**Callout 활용 가이드** — 다음 정보는 본문이 아닌 callout으로 분리합니다:
- MVP 단계의 범위 — End Picture 비전과 구분
- 핵심 기능 목록 — 가치의 수단, 흐름 방해 방지
- 지표 산출 근거·주석 — 수치의 의미·전제 설명

**포맷:**
```
# End Picture  (Impact와 --- 구분선으로 분리)

[비전 서술: 제품적 가치의 방향성 — 구체적 수치나 고객명이 아닌, 이 제품이 어떤 역할을 하게 되는지의 방향성을 1-2문장으로. 하위 섹션의 요약이 아니라 제품 철학]

## 시장 포지션
- [이 시장에서 현재 어디에 서 있는가 — 벤치마크 비유 포함]
- [어디로 확장하려는가 — 지금 포지션이 어디까지 열어주는가]

## 타겟 고객군
1. [세그먼트 1: 구체적 고객 유형]
2. [세그먼트 2: 구체적 고객 유형]
[⚠️ callout: MVP 단계의 좁은 범위 명시]

## 핵심 가치
- [세그먼트 1 니즈 → 제공 가치]
- [세그먼트 2 니즈 → 제공 가치]
[callout: 핵심 기능 목록]

## 핵심 지표 (기간 후 목표)
[왜 이 지표로 니즈 충족을 확인하는가 — 1문장]

| 지표 | 현재 | 목표 |
|---|---|---|
| ... | ... | [세그먼트 포함 목표값] |

[callout: 지표 산출 근거]

## 세일즈/운영 구조
- [세일즈: 어떻게 고객을 온보딩하는가]
- [운영: 어떤 시스템·팀으로 이 가치가 지속되는가]
```

**End Picture에서 피할 것:**
- 비전 서술에서 하위 섹션의 요약을 쓰는 것 → 비전 서술은 제품적 방향성, 목차가 아님
- 타겟 고객군에서 MVP 범위만 기술하는 것 → End Picture 비전과 MVP를 혼동하지 말 것
- 핵심 가치 bullet이 타겟 세그먼트 수와 맞지 않는 것
- 핵심 가치에서 기능을 먼저 쓰는 것 → 니즈 → 가치 → 기능(callout) 순서
- 지표에 타겟 세그먼트가 반영되지 않은 것
- Phase별 진행 계획 → Roadmap으로
- 기술 구현 상세 → 각 Phase PRD로
- "기여합니다", "향상됩니다" 같은 추상 동사로만 끝내는 것
- Impact와 내용이 겹치는 것 ("문제가 해결된 상태" ≠ "완성된 비전")

---

### Roadmap

**Epic을 구성하는 Phase 목록.** 각 Phase는 독립적으로 가치를 제공해야 합니다.

**각 Phase에 포함할 내용:**
- 1문장 요약 (이 Phase가 달성하는 것)
- **핵심 기능**: 이 Phase에서 처음 추가되는 기능 목록 — 상세 구현은 PRD가 담당하지만, 어떤 기능이 추가되는지는 Roadmap 레벨에서 보여야 함
- **타임라인**: 목표 시작·완료 시점 또는 분기 (미정이면 "미정"으로 명시)
- **진입 조건** (Phase 2 이후): 이전 Phase의 어떤 성과·산출물이 충족되어야 이 Phase를 시작하는가
- PRD 링크

**포맷:**
```
## Phase 1: [이름] — [분기 또는 월]
[1문장: 이 Phase가 달성하는 것]

**핵심 기능**
- [기능 1]
- [기능 2]

**타임라인**
- [월/이벤트]: [마일스톤]
- [월/이벤트]: [마일스톤]

PRD: [링크 또는 "미작성"]

## Phase 2: [이름] — 미정
[1문장]
*(진입 조건: [Phase 1의 어떤 조건이 충족되면 시작])*

**핵심 기능 추가** (Phase 1 대비 추가되는 것만)
- [기능 1]
- [기능 2]

**타임라인**: 미정

PRD: [링크 또는 "미작성"]

## Phase 3: [이름] — [진입 조건] 완료 후
[1문장]
*(진입 조건: [명확한 조건 — 지표 달성, 특정 기능 완성 등])*

**핵심 기능 추가**
- [기능 1]

**타임라인**: 미정

PRD: [링크 또는 "미작성"]

## Out of Scope (이번 Epic에서 제외)
- [명시적으로 제외하는 항목들]
```

**Roadmap 원칙:**
- 핵심 기능은 "이 Phase에서 처음 추가되는 것"만 기술 — 이전 Phase 기능 반복 금지
- 구현 상세(어떻게 만드는가)는 없음 → PRD가 담당
- 각 Phase는 독립적으로 배포·롤백 가능한 단위
- Phase 2 이후 타임라인은 유동적이므로 "미정" 또는 진입 조건으로 표현
- Phase 2 이후 진입 조건을 명시해 "언제 시작하는가"의 기준을 독자가 이해할 수 있게 함

## Stage 3: Self-Check

**Problem**
- [ ] 1문장, 절 2개 이하인가?
- [ ] Epic 수준의 구조적·전략적 문제인가?
- [ ] ~합니다 체로 끝맺는가?

**Background**
- [ ] 5줄 이내인가?
- [ ] 구체적 구현 지시가 없는가? (방향 언급은 허용, "~기능을 만들어야 합니다"는 금지)
- [ ] DARO 공유 컨텍스트를 재설명하지 않는가?
- [ ] 각 줄이 이전 줄에서 논리적으로 이어지는가?
- [ ] 마지막 줄이 Roadmap Phase 1으로 자연스럽게 연결되는가?
- [ ] Problem → Background → Roadmap을 순서대로 읽었을 때 "그래서 이 방향이구나"가 자연스럽게 이해되는가?
- [ ] 논리적 묶음 사이에 줄바꿈이 있어 읽는 호흡이 명확한가?
- [ ] 구체적 출발점이 사실 나열이 아니라 기회 서사로 쓰여 있는가?
- [ ] 플라이휠 역할 전환(기존 관계자의 새 역할)이 있다면 명시했는가?

**Impact**
- [ ] End Picture나 Solution 기능이 아닌, 문제가 해결된 상태인가?
- [ ] 전략적 수준의 변화를 다루는가?
- [ ] Bullet 3개 이하 또는 AS-IS/TO-BE 테이블인가?

**End Picture**
- [ ] 비전 서술이 하위 섹션 요약이 아닌 제품적 방향성인가?
- [ ] 시장 포지션에 벤치마크 비유가 포함됐는가?
- [ ] 타겟 고객군이 End Picture 전체 세그먼트를 커버하는가? (MVP 범위로 제한하지 않았는가?)
- [ ] MVP 범위가 좁다면 callout으로 분리됐는가?
- [ ] 핵심 가치 bullet 수가 타겟 세그먼트 수와 일치하는가?
- [ ] 핵심 가치에서 니즈 → 가치 → 기능(callout) 순서인가?
- [ ] 핵심 지표에 타겟 세그먼트가 모두 반영됐는가?
- [ ] 지표 산출 근거가 callout으로 분리됐는가?
- [ ] 운영 구조가 세일즈·운영을 모두 포함하는가? ("세일즈/운영 구조" 헤딩)
- [ ] Phase 계획이나 기술 구현이 섞여 있지 않는가?
- [ ] Impact와 내용이 겹치지 않는가?
- [ ] 의사결정권자가 이 그림을 보고 장기 투자를 결정할 수 있는 수준인가?
- [ ] 추상 동사("기여", "향상", "개선")로만 끝나는 문장이 없는가?

**Roadmap**
- [ ] 각 Phase가 독립적으로 가치를 제공하는가?
- [ ] 각 Phase에 핵심 기능 목록이 포함됐는가? (처음 추가되는 것만)
- [ ] Phase 1에 타임라인(마일스톤)이 포함됐는가?
- [ ] Phase 2 이후에 진입 조건이 명시됐는가?
- [ ] Phase 2 이후 타임라인이 "미정" 또는 진입 조건으로 표현됐는가?
- [ ] 구현 상세(어떻게 만드는가)가 없는가? (상세는 PRD로)
- [ ] Out of Scope가 명시되어 있는가?

**문체 (전체)**
- [ ] ~합니다 / ~습니다 체로 일관되는가?
- [ ] 확실한 사실 → ~합니다, 가능성 → ~수 있습니다로 구분되는가?
