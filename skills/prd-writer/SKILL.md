# PRD Writing Skill

A workflow for writing PRDs that engineers can actually execute from.

The four-section structure used here:

  # Purpose (Problem)
  # Background
  # Impact
  # Solution

---

## 핵심 원칙 (모든 섹션에 적용)

**문체는 ~합니다 / ~습니다 체 (경어체)**를 사용합니다.
- ✅ "운영자가 트래픽을 직접 제어할 수 있는 기능이 없어, 개발 리소스 의존이 발생하고 있습니다."
- ❌ "운영자가 트래픽을 직접 제어할 수 있는 기능이 없어, 개발 리소스 의존이 발생한다."

`~한다 / ~된다` 체(평서체)는 어떤 섹션에서도 사용하지 않습니다.
단, 코드 스니펫, 테이블 내용, 항목 레이블 등 문장이 아닌 요소는 예외입니다.

**확신의 정도에 따라 어미를 구분합니다.**
- 명확한 근거가 있거나 확실한 사실 → `~합니다 / ~됩니다`
  예: "Bid Request의 `imp.pmp` 필드에 Deal 정보를 포함합니다."
- 가능성이 높지만 확실하지 않은 경우 → `~수 있습니다 / ~될 수 있습니다`
  예: "프리미엄 DSP 유입으로 파트너십이 확대될 수 있습니다."
  예: "Deal 매칭 로직 추가로 레이턴시에 영향을 줄 수 있습니다."

---

**용어 표기 규칙 (영어/한국어 병기).**
- 영어로 더 널리 알려진 용어는 **그대로 영어로** 씁니다. 한국어로 번역해 괄호로 병기하지 않습니다.
  - ✅ "Bid Request", "waterfall", "fill-rate", "frequency capping"
  - ❌ "입찰 요청(Bid Request)", "폭포수(waterfall)"
- 괄호 병기는 **단 하나의 경우** - 전문 용어나 줄임말이 문서에서 **처음 등장할 때 1회만** 괄호 안에 풀어 씁니다. 같은 설명을 여러 번 반복하지 않습니다.
  - ✅ 최초 1회 "ILRD(Impression-Level Revenue Data)" → 이후로는 "ILRD"
  - ❌ 매 등장마다 "ILRD(Impression-Level Revenue Data)" 반복
- Purpose에서는 이 최초 정의조차 하지 않습니다. 용어 정의가 필요하면 Background에서 풉니다.

---

**구체성이 전부입니다.** 막연한 문장은 모두 구체적인 동작, 이름, 숫자로 바꿉니다.

| 나쁜 예 | 좋은 예 |
|---|---|
| "사용성을 개선합니다" | "마우스 오버 시 모든 지표가 한 번에 표시됩니다" |
| "수익이 향상됩니다" | "MREC 동영상 소재 수신 가능 → eCPM 향상 (정확한 수치는 운영 후 측정)" |
| "운영 효율이 개선됩니다" | "개발자 개입 없이 운영자가 demand 트래픽을 직접 ON/OFF할 수 있습니다" |

**AS-IS / TO-BE 구분.** Solution 섹션에서 현재 상태와 변경 후 상태를 항상 나란히 씁니다.

**모르면 솔직하게.** 아직 확인이 필요한 사항은 `** 확인 필요` 로 문서 안에 남깁니다.
리뷰어가 빈칸을 발견하는 것보다, 질문이 명시된 문서가 낫습니다.

---

## Stage 1: Context Gathering

글 쓰기 전에 아래를 파악합니다. 이미 맥락이 주어졌으면 거기서 추출하고, 빠진 것만 물어봅니다.

1. **지금 뭐가 문제인가?** 어떤 동작이 안 되거나, 없거나, 잘못 되어 있는가?
   목적 중심 프로젝트라면: 어떤 성과/목표를 달성하려 하는가?
2. **누가 영향을 받는가?** (앱 개발자, 내부 운영자, DSP/SSP 파트너, 최종 사용자 등)
3. **현재 시스템은 어떻게 동작하는가?** 코드 경로, API 이름, 관련 시스템명
4. **변경 후에는 어떻게 되어야 하는가?** 어떤 결과물을 만드는가?
5. **제약사항이 있는가?** (하위 호환성, SDK 버전, 규제, 일정)
6. **참고할 자료가 있는가?** (슬랙 스레드, 노션 문서, 외부 스펙)

슬랙/노션 링크가 있으면 직접 읽어옵니다.

---

## Stage 2: 섹션별 작성 가이드

### Purpose (Problem)

**목표**: **단 1문장**. 읽는 사람이 첫 문장에서 "아, 이걸 고치는 거구나" 또는 "아, 이걸 만드는 거구나"를 알아야 합니다.

**구조 1 — 문제 중심** (기본):
> `[주어]는 [무엇이] 없어서/안 돼서, [어떤 문제]가 발생합니다.`

**구조 2 — 목적 중심** (명확한 문제보다 목표가 앞서는 프로젝트):
> `[어떤 성과]를 달성하기 위해 [주어]에 [무엇을] 합니다.`

**좋은 예시들**:
> "DARO SDK는 단일 미디에이션 의존 구조로, 특정 미디에이션 성능 저하 시 DARO가 독자적으로 대응할 수 없습니다."

> "현재 DARO Ads SDK의 MREC은 이미지 소재를 받을 구조가 없어 동영상 입찰이 불가합니다."

**나쁜 예** (절이 3개 이상 연결 — Background 내용이 Purpose에 섞인 구조):
> ~~"현재 DARO SDK는 퍼블리셔가 AdMob 또는 AppLovin 중 하나에만 의존하는 분리 구조로, 단일 미디에이션 성능 변동·버그 발생 시 DARO가 독자적으로 대응할 수 없어 고객사 불만이 누적되고 수익 변동성이 통제 불가한 상태입니다."~~

→ 절이 3개 연결됨. "분리 구조 설명"과 "고객사 불만 누적"은 Background에서 다룹니다. Purpose는 핵심 문제 하나만 남깁니다.

(MREC, MRAID 등 용어 정의는 Purpose가 아니라 Background에서 풉니다.)

**체크**:
- 반드시 **1문장**. 문제가 여러 개여도 Purpose는 1문장으로 묶습니다. 상세 나열은 Background에서
- **절은 2개 이하**. 주절 + 종속절로 끝냅니다. 절이 3개 이상 연결되면 Background 내용이 섞인 것입니다. 초과분은 잘라냅니다.
- **Purpose와 Background는 내용이 겹치지 않습니다.** Purpose에 쓴 문장을 Background에서 다시 설명하거나 반복하지 않습니다. Purpose는 핵심 문제 한 줄, Background는 그 문제가 왜 존재하는지의 맥락입니다.
- **Purpose에서는 용어를 설명하지 않습니다.** 괄호를 사용한 약어/용어 정의(`MREC(300x250)` 같은)를 쓰지 않습니다. 정의가 필요하면 Background에서 풉니다. 한 호흡에 읽히는 것이 괄호 설명보다 우선입니다.
- "개선합니다", "향상시킵니다" 같은 추상 동사로 끝내지 않습니다
- 확실한 사실은 `~합니다`, 가능성은 `~수 있습니다`로 구분합니다

---

### Background

**목표**: 이 문제가 왜 존재하는지, 현재 시스템이 어떻게 동작하는지를 **5줄 이내**로 설명합니다.
맥락을 전혀 모르는 엔지니어가 읽어도 이해할 수 있어야 합니다.

**본문 (5줄 이내)**에 포함할 것:
- 현재 시스템/코드의 핵심 동작 방식
- 현재 구조 때문에 발생하는 문제의 본질
- 관련 외부 스펙이나 표준이 있다면 한 줄 언급

**토글 안에 넣을 것** (읽는 사람이 선택적으로 볼 상세 정보):
- 관련 코드 경로, API 이름, 네트워크별 매핑 테이블
- 역사적 맥락, 의사결정 히스토리
- 외부 스펙 상세 (oRTB, MRAID 등)
- 스크린샷, 다이어그램

**포맷 예시**:
```
# Background
현재 DARO Ads SDK는 RTB 방식으로 DSP와 직접 연동해 광고를 수신합니다.
미디에이션 방식과 달리 소재 수신/렌더링을 SDK가 직접 구현해야 하며,
지원하지 않는 소재 타입은 DSP 입찰 자체가 들어오지 않습니다.
현재 MREC은 이미지 소재만 구현되어 있어 동영상 소재를 받을 구조가 없습니다.

<details>
<summary>미디에이션 vs RTB 방식 비교</summary>
  ...상세 내용...
</details>
```

**피할 것**:
- Background가 Purpose의 재탕이 되는 것
- 5줄을 초과하는 본문 → 초과분은 토글로
- **"DARO 공유 컨텍스트" 섹션에 있는 전제(RTB 구조, SSP/DSP 역할, 공유 용어 등)를 재설명하는 것.** 독자는 이미 알고 있습니다. Background는 이번 PRD에만 해당하는 상황·원인만 담습니다.

---

### Impact

**목표**: 이 작업이 완료됐을 때 무엇이 달라지는가.

**두 축으로 작성**:
1. **기능적 변화**: 사용자/개발자 입장에서 새로 가능해지는 것
2. **비즈니스/시스템 변화**: 수익, 안정성, 운영 효율, 기술 부채 관점

**측정 가능하면 숫자를 씁니다.** 숫자가 없으면 방향이라도 명시하고, 측정 방법을 언급합니다.
> "동영상 소재는 이미지 대비 eCPM이 높은 것이 일반적이나, 정확한 증가폭은 실제 운영 후 데이터로 측정이 필요합니다."

**절대 쓰면 안 되는 표현들**:
- "전반적인 품질 향상"
- "사용자 경험 개선"
- "운영 효율화" (이것만으로 끝내면 안 됩니다. 구체적으로 어떤 작업이 얼마나 빨라지는지 써야 합니다)

---

### Solution

**목표**: 담당 엔지니어가 추가 질문 없이 구현을 시작할 수 있는 수준의 상세도.

**첫 문장 필수**: 섹션 시작을 반드시 1문장 요약으로 시작합니다.

> 구조: `[어떤 결과물]을 통해 [어떤 문제]를 해결합니다.`

예시:
> "MREC Bid Request에 MRAID 지원 선언을 추가하고 동영상 소재 렌더링 경로를 구현해 동영상 입찰 수신 불가 문제를 해결합니다."
> "Demand/Supply 단위 트래픽 제어 기능과 어드민 UI를 제공해 개발자 의존 없이 운영자가 RTB 트래픽을 직접 제어할 수 있습니다."

**이후 구조 (복잡도에 따라 선택)**:

단순한 작업:
```
## Solution
[1문장 요약]

### 변경 내용
- AS-IS: 현재 동작
- TO-BE: 변경 후 동작

[코드 스니펫 또는 테이블]
```

복잡한 작업 (Phase 분리):
```
## Solution
[1문장 요약]

### Phase 1: [이름]
#### 1-1. [세부 항목]
...

### Out of Scope
- 이번 작업에서 제외되는 것들
```

**자주 쓰이는 포맷들**:

AS-IS / TO-BE 코드 비교:
```javascript
// AS-IS
"banner": { "width": 300, "height": 250 }

// TO-BE
"banner": { "width": 300, "height": 250, "api": [3, 5] }
```

변경 전후 테이블:
| | AS-IS | TO-BE |
|---|---|---|
| 소재 타입 | 이미지만 | 이미지 + MRAID 동영상 |
| api 필드 | 없음 | [3, 5] 추가 |

**확인 필요 표기**: 구현 중 확인이 필요한 내용은 `** 확인 필요` 로 인라인에 명시합니다. 숨기지 않습니다.

## Stage 3: Self-Check

**Purpose**
- [ ] 1문장인가?
- [ ] 절이 2개 이하인가? (주절 + 종속절. 3개 이상이면 잘라낸다)
- [ ] 한 호흡에 읽히는가? (괄호·부연으로 호흡이 끊기지 않는가)
- [ ] Purpose에 쓴 내용이 Background에 반복되지 않는가?
- [ ] 괄호를 사용한 약어/용어 정의가 없는가? (정의는 Background로)
- [ ] "개선", "향상" 같은 추상 동사로 끝나지 않는가?
- [ ] 문제 중심이면 `[주어]+[없어서/안 돼서]+[문제]` 구조인가?
- [ ] 목적 중심이면 `[성과]+[주어]+[무엇을]` 구조인가?
- [ ] ~합니다 / ~수 있습니다 체로 끝맺는가?

**Background**
- [ ] 본문이 5줄 이내인가?
- [ ] 상세 정보는 토글 안에 있는가?
- [ ] AS-IS 동작이 명확하게 설명되어 있는가?

**Impact**
- [ ] 측정 가능한 변화 또는 측정 방법이 1개 이상 있는가?
- [ ] "전반적 개선" 류의 문장이 없는가?

**Solution**
- [ ] 첫 문장이 `[결과물]+[문제 해결]` 구조 1문장인가?
- [ ] Out of Scope가 명시되어 있는가?
- [ ] 확인 필요 사항이 숨겨지지 않고 드러나 있는가?
- [ ] 엔지니어가 이것만 보고 구현을 시작할 수 있는가?

**문체 (전체)**
- [ ] ~합니다 / ~습니다 체로 일관되게 작성되었는가?
- [ ] ~한다 / ~된다 체(평서체)가 섞이지 않았는가?
- [ ] 확실한 사실은 ~합니다, 가능성은 ~수 있습니다로 구분되었는가?

**크기 체크**: 이 PRD 하나가 Linear 티켓 1~3개로 분해 가능한가? 너무 크면 Phase 분리 또는 쪼갤 것을 제안합니다.

---

## Stage 4: Notion 작성

노션 페이지 URL이 주어지면:
1. `notion-fetch`로 현재 상태 확인
2. 빈 페이지 → `replace_content`로 전체 작성
3. 기존 내용 있음 → 섹션별로 `update_content` 사용
4. DB 페이지인 경우 `Status`를 `Grooming(To Review)`로 업데이트

## DARO 공유 컨텍스트 (Background에서 재설명 금지)

아래는 DARO 팀이 이미 공유하고 있는 전제입니다. **PRD 독자(내부 개발자·운영자)는 RTB와 DARO 구조를 이미 이해하고 있으므로, 이 섹션의 내용을 Purpose/Background에서 다시 설명하지 않습니다.** Background에는 "이번 PRD에만 해당하는 상황과 원인"만 적습니다.

예외: 아래 용어집에 없는 새롭거나 specific한 개념이 등장하면, 그때만 Background에서 간단히 정의합니다 (Purpose에서는 어떤 경우에도 정의하지 않습니다).

### DARO 한 줄 정의와 현재 가치

- DARO는 DelightRoom의 인앱 광고 수익화 플랫폼입니다.
- 현재 핵심 가치는 **미디에이션 최적화**입니다 (크로스 퍼블리셔 데이터 + ML 기반). 단일 퍼블리셔가 자력으로 도달할 수 없는 수익을 만들어주는 것이 트랙 레코드이며, 고객사 광고 매출을 현 시점 기준 평균 +80% 수준까지 증분시켜 왔습니다.
- 국내 논게임 광고 수익화 시장의 선두 사업자급이며, invitation-only 프리미엄 · 인바운드 중심으로 운영합니다.
- DARO의 수익화 노하우는 자사 논게임 앱 **Alarmy(알라미)** 운영 경험에 기반합니다.

### DARO 제품 아키텍처 (현재 구조) - **반드시 이해할 것**

DARO는 미디에이션을 **대체**하는 것이 아니라, 미디에이션을 **베이스로 그 위에 최적화 레이어와 자체 RTB를 얹는** 구조입니다.

- **베이스 = AdMob / AppLovin MAX 미디에이션.** DARO의 광고 노출은 이 두 미디에이션 위에서 동작합니다.
- 두 미디에이션의 **Management API + Report API를 연동**해, 운영·리포팅을 **DARO 자체 대시보드**로 통합 제공합니다.
- 미디에이션과 그 산하 애드네트워크들을 하나로 묶은 **통합 SDK = DARO SDK**를 퍼블리셔에게 제공합니다 (복잡한 SDK 통합을 DARO가 대행).
- 여기에 더해, DARO **자체 RTB 애드서버**를 **Daro Ads SDK**와 함께 제공하며, 이 RTB 애드서버를 **미디에이션 안에 하나의 demand source로 포함**시킵니다. 즉 DARO의 oRTB는 미디에이션 워터폴/비딩을 대체하지 않고, 그 안에 들어가는 **추가 디맨드 라인**입니다.
- 이 RTB 애드서버는 SSP로서 DSP에 **Bid Request**를 보내고 **Bid Response**를 받아 경매로 win을 결정합니다. SSP인 DARO는 conversion postback(MMP)을 직접 전송할 수 없습니다 (구조적 제약).
- 프로토콜은 OpenRTB 2.x 표준을 따르며, 2.6 요소(예: `imp.rwdd`)를 도입하고 있습니다.

### DARO 제품 전략 3축 (Impact 프레이밍에 활용)

기술력으로 "퍼블리셔가 단독으로 할 수 없는 영역"을 점령하는 것이 대체 불가능성의 원천입니다. 세 축으로 정리됩니다.

| 축 | 정의 | 예시 |
|---|---|---|
| **Demand 극대화** | 퍼블리셔 단독으로 확보 불가한 수요 | oRTB 디맨드, 직광고(GAM & DSP) |
| **Yield 극대화** | 퍼블리셔 단독으로 도달 불가한 단가 | PMP, Prebid, ML 기반 최적화 |
| **Quality 보호** | 퍼블리셔 단독으로 유지 불가한 지면 품질 | 소재 필터링, 하이재킹 방어, 정책·개인정보 대응 |

- **RTB 애드서버 고도화 · PMP · DSP**는 모두 미디에이션만으로는 낼 수 없는 **"독자 디맨드"와 eCPM 경쟁력**을 확보하기 위한 수단입니다. 이는 게임·글로벌 시장 확장의 기술적 전제이기도 합니다 (게임은 미디에이션 활용도가 높아, 독자 디맨드 없이는 차별적 eCPM을 제시하기 어렵습니다).
- 장기 방향은 유통 구조 흡수입니다: 미디에이션 최적화 레이어 → SSP로 유통 단계 축소 → DARO DSP 추가로 광고 수익이 생태계 안에서 순환하는 플라이휠.

> **주의**: 위 매출 증분(+80%)·시장 순위·수치는 작성 시점 기준의 컨텍스트이며, 시간이 지나면 변합니다. PRD 본문에 고정 사실처럼 인용하지 말고, 필요하면 최신 수치를 확인하거나 전략 문서를 참조하세요.

> **참조 문서** (컨텍스트가 더 필요할 때 Notion에서 직접 읽기): "26-27 다로 전략 정리"(상위), 하위에 "제품 확장 전략" / "게임 시장 GTM 전략" / "세일즈 & 운영 전략".

### 공유 용어집 (재설명 불필요)

| 용어 | 의미 |
|---|---|
| oRTB / RTB / OpenRTB | 프로그래매틱 광고 실시간 경매 프로토콜 |
| SSP / DSP | 공급(퍼블리셔) 측 / 수요(광고주) 측 플랫폼 |
| Demand / Supply | 광고 수요자(DSP) / 공급자(퍼블리셔 앱) |
| Bid Request / Bid Response | 경매 요청 / DSP의 응찰 |
| `crid` | 소재(creative) 식별자. DARO는 `dsp_id`+`crid` 조합으로 식별 |
| `adomain` | 광고주 랜딩 도메인 (배열) |
| `bundle` | 광고주 앱 번들/스토어 패키지 ID |
| `cat` | IAB 콘텐츠 카테고리 |
| `iurl` / `adm` | 소재 샘플 이미지 URL / 광고 마크업 |
| `nurl` / `lurl` / `burl` | win notice / loss notice / billing notice URL |
| eCPM | 1,000회 노출당 실질 수익 |
| bid rate / win rate | 응찰률 / 낙찰률 |
| ILRD | Impression-Level Revenue Data. 노출 단위 수익 데이터 (미디에이션 reward signal) |
| Waterfall / Mediation | 광고 네트워크 순차 호출 / 통합 관리 방식 |
| Banner / Native / MREC | 기본 광고 포맷 (MREC = 300x250) |
| Interstitial / Rewarded Video | 전면 / 보상형 동영상 포맷 |
| VAST / MRAID | 동영상 광고 표준 / 모바일 리치미디어 표준 |
| Deal ID / PMP | Private Marketplace. 특정 DSP에만 허용하는 비공개 경매 |
| AdMob / AppLovin MAX | DARO의 베이스 미디에이션 플랫폼 |
| Management API / Report API | 미디에이션 설정·리포팅 API. DARO 대시보드의 데이터 소스 |
| DARO SDK / Daro Ads SDK | 미디에이션+애드네트워크 통합 SDK / DARO 자체 RTB 애드서버용 SDK |
| GAM / AdX / Open Bidding | Google 광고 서버 / 거래소 / 입찰 방식 |
| SKAdNetwork | Apple의 어트리뷰션 프레임워크 |
| sellers.json / app-ads.txt | 공급망 투명성 표준 |
| IAA / IAP | 인앱 광고 수익 / 인앱 구매 수익 |
| fill-rate / match-rate | 광고 채워지는 비율 / 요청-응답 매칭률 |
| frequency capping | 동일 유저 광고 노출 빈도 제한 |
| AXON | AppLovin의 머신러닝 입찰/UA 엔진 |
| GDPR / CCPA / COPPA | 개인정보 보호 규정 (유럽 / 미 캘리포니아 / 아동) |
| T1 / T2 | 퍼블리셔 규모 분류. T1: 상위권, T2: 중위권 |
| ACV | Annual Contract Value. 연간 계약 금액 |
| NRR | Net Revenue Retention. 순 매출 유지율 |
